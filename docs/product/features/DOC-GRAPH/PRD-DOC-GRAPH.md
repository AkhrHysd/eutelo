# PRD-EUTELO-GRAPH

## Background

Eutelo はドキュメントを中心とした開発体系であり、PRD / BEH / DSG / ADR / TASK など
複数種別の文書が相互に関係し合うことで、プロダクトの仕様や計画を支える。

しかし現状では、この関係性は主に以下のように「暗黙的」または「人間依存」で保持されている。

- 「PRD を変えたらどの BEH が影響するか」は、経験者が記憶に基づいて判断している  
- 新規ドキュメント追加時に「何と紐づけるべきか」が明示的に見えない  
- 全体的な体系（どの機能にどれだけ文書があるのか）が把握しづらい  

この状態は、プロジェクト規模が一定を超えると破綻しやすい。

そこで、**ドキュメントを体系として扱い、関係性を可視化する概念レイヤ**が必要になる。

## Goals / 成し遂げたいこと

1. **ドキュメント群を「点」ではなく「ネットワーク」として捉えられる状態をつくること。**  
2. **仕様変更や追加の際に、波及する可能性のある範囲を即座に把握できるようにすること。**  
3. **合意された構造・関係性を、時間とともに維持しやすくすること。**  
4. **新規参画者が、プロジェクトの仕様体系を理解しやすくなる状態を作ること。**

これらは「実装手段」ではなく「価値」そのものとして定義する。

## Non-Goals / やらないこと（境界）

- 完全な自動推論（LLMで暗黙的な関係を自動生成するなど）
- ドキュメントの内容そのものを評価する（意味評価は別機能）
- Web UI を含む可視化プラットフォームの提供
- 依存関係が正しいかどうかの判定（それはガードレール・LLMが担う）

Eutelo Graph は **「関係性を構造として提示する」** ことに責務を限定する。

## What / この機能が提供するもの

- **ドキュメント間の関係性を“構造として理解可能な形”で取り出せる能力**
- **任意の文書を起点に、その前後関係（上位/下位/関連）を把握する能力**
- **変更に対して、潜在的に影響する範囲を素早く列挙する能力**
- **ドキュメント体系の全体像（俯瞰）を、抽象レベルで提示する能力**

いずれも「ドキュメントそのものの意味」には踏み込まない。

## Why / この機能が必要な理由

1. **人間の記憶・認知に依存した仕様管理はスケールしないため。**  
2. **仕様の整合性は、個々の文書ではなく文書“間”の関係に宿るため。**  
3. **PRD/BEH/DSG/ADR の責務分離を行うと、関係数が急激に増え、頭で管理できなくなるため。**  
4. **体系を維持するためには「俯瞰できる構造物」が必要になるため。**

Eutelo Graph は、Eutelo 全体の「脳内地図」のような役割を果たす。

## Example Scenarios / ユースケース（抽象レベル）

### 1. PRD を更新する開発者
- 機能改修に伴い PRD を更新した時、  
  **どの振る舞い・設計が影響するか** を迷わず確認したい。

### 2. レビューア
- PR 内で仕様が変わった際、  
  **関連するドキュメントの修正漏れ** を素早く把握したい。

### 3. 新規メンバー
- プロダクト全体の構造を理解したいが、  
  **どこからどこまでが同じ機能領域なのか** が直感的に分からない。

### 4. プロダクトオーナー
- 機能群としてのドキュメントの偏り（未完領域や過密領域）を  
  **構造として把握したい**。

## Success Indicators / 成功の判断軸（KPIではなく抽象指標）

- **仕様変更時の修正漏れが体系的に減ること**  
- **新規参画者のキャッチアップ速度が向上すること**  
- **ドキュメント体系の“全体地図”が会話の前提として共有されること**  
- **レビュー基準が“属人の記憶”から“構造に基づく判断”に変わること**

## Scope（抽象）

### In Scope（この機能が扱うもの）
- ドキュメント間の「関係構造」を抽象的に把握・提示すること  
- 体系としての「つながり」の定義と抽出  
- 「変更が生み出す影響領域」の概念モデル化

### Out of Scope（扱わないもの）
- 関係の正しさの審査（それはガードレール領域）  
- 意味の矛盾の検出（それはLLM領域）  
- リッチUIやダッシュボードの提供  
- 暗黙的関係の推論（次フェーズ以降）

## Requirements（抽象版）

### Functional
- ドキュメント体系の「関係構造」を取得できる状態を提供する  
- 任意のドキュメントから「隣接する関係」を辿れる状態を提供する  
- 変更時に関係構造を用いて「潜在的影響領域」を明確にできる  
- 関係構造を周辺サービス（CLI / CI / 他機能）から利用可能にする

### Non-Functional
- プロジェクト規模に応じて扱える柔軟なモデルであること  
- 文書体系が変化しても破綻しない構造抽象であること  
- 他機能（ガードレール / 統治モジュール）と干渉しないこと

## Risks（抽象）

- 文書間の関係表現が不十分な場合、構造グラフが本質を反映しない  
- 人間の認知モデルと、構造モデルの乖離が起きる可能性  
- 「関係の正しさ」を表現すべき境界が曖昧になりやすい（責務の分離が重要）
